01 — Overview

What was built

MarkControl is a full-stack feature flag control plane that decouples code deployment from feature release. It lets engineering teams ship continuously while controlling who sees what — per user, per org, per plan — with no redeploys required.

⚡

In-memory evaluation

Rules are loaded from DB (or 60s TTL cache) and evaluated entirely in Python — no DB hit per flag check.

🎯

5-level priority chain

User → Org → Plan → Attribute → Percentage → Default. Deterministic, documented as a constant.

🔒

Separate auth layers

Public /evaluate endpoint is read-only. Admin endpoints require x-admin-token header.

📋

Full audit trail

Every create, update, and rule mutation is persisted with actor, timestamp, old state, and new state JSON.

02 — Architecture

System architecture

The system follows a strict layered design. The React frontend holds no evaluation logic — it is a pure consumer of the /evaluate endpoint, which acts as the single source of truth for both sides of the stack.

System overview — request flow

Data model

The schema is designed around the separation of mark identity from targeting logic. A Mark record holds the key, type, and default. Rules are a separate table allowing N rules per mark, each independently toggled.

03 — Evaluation Engine

The core engine (D2)

The evaluation engine is a pure Python module with zero web framework dependencies. It receives a Mark, a list of MarkRule objects, and an EvaluationContext — and returns a bool | str. The priority order is declared as a constant, not buried in conditionals.

Evaluation priority — decision flow per request

Each rule type also supports an optional percentage sub-rollout column — enabling intersectional logic like "50% of Org A" without building a full AST parser. 

# constants.py — priority is a first-class constant, not implicit logic
RULE_PRIORITY = [
    "user",        # most specific — canary releases, QA targeting
    "org",         # multi-tenant isolation
    "plan",        # subscription-tier gating
    "attribute",   # flexible JSON subset matching
    "percentage",  # deterministic A/B rollout
    "default"      # safety net fallback
]

Deterministic percentage rollout

Rather than storing per-user bucket assignments (which would scale to millions of rows), the engine uses a stateless hash function. The same user_id string always hashes to the same bucket — guaranteeing sticky assignment with zero state management.

MD5 hash rollout — why it is deterministic

TTL cache in MarkClient

The MarkClient wraps the engine with a 60-second in-process TTLCache. Every database read is gated behind a cache key of the form mark_data:<key>. The cache is explicitly invalidated on any admin mutation, and fully cleared on cascading operations.

The engine module has zero imports from FastAPI, SQLAlchemy, or any HTTP library. It takes plain Python objects and returns a plain value — making it independently testable without a running server, exactly as required by the spec.

04 — API Layer (D1)

Evaluation & admin API

Method Endpoint Auth Purpose
POST /api/v1/marks/evaluate None (public) Batch evaluate N marks for a context. Returns { "key": bool } map. Silent-fails to empty object on error.
GET /admin/marks x-admin-token List all marks with type, state, rule count, parent_id, tags.
POST /admin/marks x-admin-token Create a new mark with key, type, default_value, optional parent_id, tags.
PATCH /admin/marks/{id} x-admin-token Toggle is_active (with optional cascade), update parent, tags, default. Writes audit log.
POST /admin/marks/{id}/rules x-admin-token Add a targeting rule. Prevents duplicate identical rules.
PUT /admin/marks/{id}/rules/bulk x-admin-token Transactional sync of all rules — diffs existing vs incoming, creates/updates/deletes in one commit.
DELETE /admin/marks/{id}/rules/{rule_id} x-admin-token Remove a single rule.
GET /admin/marks/{id}/audit_logs x-admin-token Last 50 audit entries for a mark, descending by timestamp.
DELETE /admin/marks/{id} x-admin-token Delete mark (blocked if children exist).

Evaluate request shape

# POST /api/v1/marks/evaluate
{
  "user_id": "user_1",
  "org_id":  "org_a",
  "plan":    "pro",
  "attributes": { "country": "US", "device": "mobile" },
  "mark_keys": ["new-dashboard", "billing-kill-switch"],
  "tags":      ["dashboard", "experiment"] // resolves all marks with these tags
}

# Response — all requested marks in one round-trip
{
  "new-dashboard":       true,
  "billing-kill-switch": true,
  "beta-feature":        false
}

Silent degradation

The evaluate endpoint wraps its entire body in a try/except. Any unhandled exception returns an empty {} dict — the frontend then falls back to its localStorage cache or hardcoded defaultValue. The mark service going down is a silent degradation, never an application crash.

Hierarchical kill switches with cascade

Parent-child relationships are expressed through the parent_id foreign key on marks. When an operator toggles a parent's is_active with cascade: true, the backend recursively walks the tree and physically updates all children's is_active column. The cache is fully cleared afterward.

Design choice: Cascading updates the database state rather than dynamically evaluating parent rules at runtime. This keeps mark state predictable — a product manager can look at any toggle in the admin console and know its exact state without reasoning about parent rule logic.

05 — Frontend (D3)

React context provider & hook

MarkProvider lifecycle — single fetch, zero-latency hook reads

useMark hook

Any component calls useMark("feature-key", defaultValue) — it reads from the in-memory React Context with zero network overhead. If the key is not in the context (service was down), it falls back to the provided defaultValue. 

// Three gated components from page.tsx

// 1. Release mark — org-targeted
const newDashboard = useMark("new-dashboard", false);

// 2. Kill switch
const billingEnabled = useMark("billing-kill-switch", true);

// 3. A/B experiment
const betaFeature = useMark("beta-feature", false);

// The provider loads all tags in one request at layout level:
<MarkProvider tags={["dashboard", "experiment", "billing", "checkout"]}>

The MarkProvider also accepts a setContext function, enabling the demo page's live context switcher — users can change user_id, org_id, and plan to watch mark evaluations update in real-time.

06 — Admin Console (D4)

Operator interface

The Admin Console is a high-density single-page app at /admin. It follows a file-explorer pattern: a left sidebar (mark registry tree) and a right detail pane.

🌲

Hierarchical tree sidebar

Marks are displayed as an expandable tree, reflecting their parent-child relationships via TreeItem recursion.

📝

Draft-based editing

All changes are accumulated in draftRules and draft state. A single "Save Rules" call sends a PUT /rules/bulk transaction — no network request per keystroke.

⚡

One-click kill switch

Toggle in the header bar flips is_active in a single PATCH call. No confirmation dialog — speed is the point during an incident.

📜

Audit log timeline

Right column shows the last 50 audit entries per mark with action, actor, and timestamp as a visual timeline.

Bulk rule sync design

Rather than calling the API on every rule change, the frontend maintains a draftRules array in local state. When "Save Rules" is clicked, a single PUT /admin/marks/{id}/rules/bulk call diffs the full rule list against the database — creating new rules, updating changed ones, and deleting removed ones, all in one database transaction.

New rules added locally are assigned a temp-{timestamp} ID. The bulk endpoint detects these by the temp- prefix and creates them as new rows, while numeric IDs trigger updates.

07 — Engineering Decisions

Challenges & solutions

Challenge 1 — OR vs AND targeting (Intersectional sub-rollouts)

Early designs treated all rules as an OR list. An org match would immediately grant access, bypassing percentage rollout rules entirely. This made "50% of Org A" impossible to express.

Sub-rollout — how "50% of Org A" is modelled

Tradeoff: This provides ~90% of the flexibility of a full AST expression engine at 1% of the latency cost. An AST parser would increase evaluation from ~2ms to ~20ms+ — unacceptable for a service called on every page load.

Challenge 2 — Hierarchical kill switch design

The initial approach recursively evaluated parent rule chains at runtime. This had a critical flaw: a parent with no rules defaulted to false, which silently killed all children even when those children had active targeting rules.

Solution: Hierarchy cascade is explicit state, not runtime evaluation. When an operator toggles a parent with cascade: true, the backend physically writes is_active = false to all children. The state is what the database says it is — no inference required.

Parent → child cascade — physical state, not dynamic evaluation

Challenge 3 — Decoupling identity validation

Early versions queried the memberships table on every /evaluate call to verify the user_id belonged to the org_id. This created two problems: it hit the database on every flag evaluation, and it tightly coupled the Control Plane to the application's membership business logic.

Solution adopted: The evaluation engine now trusts the EvaluationContext payload it receives. Authentication and authorization must happen before the feature flag call — in the application's own auth middleware. The Control Plane becomes a stateless calculator that any service can call without being coupled to its user model.

Known tradeoff: This means a crafted request with org_id: "org_a" from a user who is not a member of org_a could evaluate org-targeted flags correctly. The spec acknowledges this and places the enforcement responsibility upstream. The membership table and the commented-out validation code remain in the codebase for operators who want to re-enable it.

08 — Design Questions

Q&A

Q1 — A mark is enabled for Org A but disabled globally. What does a user in Org A see? What does a user in no targeted org see?

The engine evaluates rules in priority order. For a user in Org A, it reaches step ② (ORG RULE) and finds a matching rule with rule_value == "org_a" and no sub-rollout. It immediately returns true — the global default is never reached.

For a user in no targeted org, the engine steps through User → Org → Plan → Attribute → Percentage checks. Finding no matching rule, it falls to step ⑤ and returns mark.default_value — which in this case is false ("disabled globally"). The org rule's true result was scoped to org_a's users only.

Q2 — User X and User Y are both in Org A with a 50% rollout. Can they see different values?

Yes, and this is by design. The hash function operates on user_id, not org_id. User X and User Y may hash to different buckets (e.g. bucket 37 and bucket 62) — one falls below the 50% threshold and gets true, the other does not.

Per-user consistency is the correct default. In A/B testing, the goal is to split individuals, not organizations. Splitting by org would mean an entire org sees one variant, collapsing the experiment sample size and introducing organizational confounders. Per-user hashing guarantees that each individual's experience is sticky across sessions, days, and requests — without any stored state.

Q3 — How do you enable a mark for every Pro plan user across all orgs, without adding one rule per org?

Add a single rule with rule_type = "plan" and rule_value = "pro". The mark_rules table stores rule type as a discriminator column — a plan rule is not scoped to any org. When the engine reaches step ③ (PLAN RULE), it checks rule.rule_value == ctx.plan. Any user whose EvaluationContext.plan field is "pro" — regardless of their org — matches this rule and receives the enabled variant.

This is why plan is a first-class rule type in the constants rather than a custom attribute — it enables cross-org targeting in a single row, as opposed to the O(orgs) duplication that adding one rule per org would require.

Q4 — Where exactly is org isolation enforced, and how do you test it holds?

Org isolation has two layers. First, at the rule-storage layer: an org rule's rule_value is the org identifier (e.g. "org_a"). The engine only returns true for that rule when ctx.org_id matches — a user in org_b cannot match an org_a rule by construction, regardless of what mark key they query.

Second, at the payload-trust layer (as discussed in Challenge 3): enforcement is upstream. The application's auth middleware must verify that the org_id in the EvaluationContext is the org the authenticated user actually belongs to. The commented-out membership check in marks.py is available for operators who want to enforce this at the control plane layer as well.

The integration test covers this: the seed data has org_a with an active org rule on new-dashboard. Evaluating with org_id = "org_a" returns true; evaluating with org_id = "org_b" returns false (the default). These are distinct assertions on the same endpoint.

Q5 — A mark was changed at 2am and caused an incident. How does the audit log answer "who, what before, what after" in under 60 seconds?

Every mutation — mark creation, toggle, rule add/update/delete — writes an AuditLog row with: mark_id, actor (the authenticated admin token identity), timestamp (server-side func.now()), action (e.g. UPDATE_MARK), and JSON-serialized old_state and new_state snapshots.

The on-call engineer calls GET /admin/marks/{id}/audit_logs (or reads the Admin Console's Audit Log timeline). The last 50 entries are returned sorted descending by timestamp. The 2am entry shows old_state: {"is_active": true} → new_state: {"is_active": false} with the actor and timestamp. Recovery is a single PATCH call to set is_active: true. Total time: under 60 seconds.

09 — Tests & Infrastructure (D5)

Tests, seed data & Docker

Unit tests

Tests live in app/tests/test_engine.py and use plain Python SimpleNamespace objects — no database, no FastAPI, no HTTP client. The engine module's zero-dependency design makes this possible.

Test What it verifies
test_user_rule_priority User rule fires before org rule when both match. Confirms step ① beats step ②.
test_percentage_rollout_is_deterministic Same context evaluated twice returns identical result. Confirms hash stability.
test_inactive_mark_returns_false A mark with is_active=False returns false regardless of rules or default_value.
test_default_fallback No matching rules → default_value is returned.

Seed data

The seed.sql file creates a representative dataset: 2 orgs (org_a, org_b), 4 users distributed across them, and 5 marks covering all major types:

Mark Type Rule Expected behavior
new-dashboard release org = org_a true for org_a users, false for org_b
beta-feature experiment percentage = 50% deterministically true for ~half of users
billing-kill-switch kill_switch user = user_4 true only for user_4 (other users get default=true)
checkout-system kill_switch (none) default = true (parent)
checkout-v2-ui release org = org_a, parent = checkout-system true for org_a; false when parent toggled off with cascade

Docker Compose

A single docker-compose up starts all three services. The backend defaults to SQLite locally but switches to Postgres via the DATABASE_URL environment variable in production. 

# Three services — one command
services:
  postgres:   port 5432, seeds from seed.sql on init
  backend:    port 8000, depends_on postgres, DATABASE_URL injected
  frontend:   port 3000, depends_on backend
10 — Future Roadmap

Production hardening

Alembic migrations

Replace create_all with tracked migration scripts for deterministic schema evolution across environments.

Redis + SSE

Distributed cache across Uvicorn workers, and Server-Sent Events so frontend clients receive push updates when a flag toggles — no page refresh needed.

DAG cycle detection

Full BFS/DFS traversal on parent assignment to prevent deep circular dependencies (A → B → C → A) which would cause infinite recursion in hierarchy logic.

Multi-environment

Add environment_id (dev/staging/prod) to marks and rules — allowing 100% rollout in staging with 0% in production on the same identity key.

Audit log visualization

History tab rendering audit logs as a visual timeline, enabling SOC2 compliance workflows and change tracking per flag.

SDK clients

Typed Python and Node.js SDK packages wrapping the MarkClient pattern, making integration into any microservice a one-liner.
